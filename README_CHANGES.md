# Опис інфраструктурних змін (Infrastructure & DevOps Update)

Цей документ містить детальний опис усіх змін, внесених у проєкт для забезпечення його відповідності вимогам розгортання в хмарі Azure за допомогою Kubernetes.

## 1. Контейнеризація (Docker)
- **Крос-компіляція:** У всіх `Dockerfile` (`data_load`, `data_quality_analysis`, `data_research`, `visualization`, `web`) додано інструкцію `FROM --platform=linux/amd64`. Це дозволяє коректно збирати образи на локальній машині з ARM64 (macOS M1/M2) для запуску на Linux VM (AMD64) в Azure.
- **Оновлення Docker Compose:** Файл `compose.yaml` оновлено для підтримки специфікації платформи (`platform: linux/amd64`) та правильної структури контекстів збірки. Це гарантує ідентичність локального середовища та хмарного.

## 2. Інфраструктура як код (Terraform)
Створено повноцінну конфігурацію в папці `terraform/`:
- **Azure Resources:** Автоматичне створення Resource Group, VNet, Subnet, Public IP та Virtual Machine.
- **Безпека (NSG):** Налаштовано правила доступу для наступних портів:
  - `22` (SSH) — для віддаленого керування.
  - `6443` (Kubernetes API) — для kubectl.
  - `30001` (Web Application) — основний інтерфейс.
  - `30080` (Argo CD) — інтерфейс керування деплоєм.
  - `30000` (Grafana) — моніторинг.
  - `30090` (Prometheus) — метрики.
- **Cloud-init:** Сценарій `cloud-init.yml` автоматизує встановлення Docker, Docker Compose та необхідних системних залежностей відразу після створення VM.

## 3. Оркестрація (Kubernetes / k3s)
Створено папку `k8s/` з маніфестами для розгортання:
- **Namespace:** Виокремлено середовище `open-data-ai`.
- **Database:** Postgres розгорнуто як Deployment з PersistentVolumeClaim для збереження даних.
- **App Pipeline:** Основний застосунок розгорнуто через механізм `initContainers`. Це дозволяє виконувати аналітичні скрипти (`data_load`, `quality`, `research`, `visualization`) послідовно та гарантувати, що веб-сервер стартує лише після готовності всіх звітів та графіків.
- **NodePort Services:** Всі ключові сервіси доступні через фіксовані NodePort для зручності без необхідності складного Ingress на етапі прототипу.

## 4. CI/CD (Argo CD)
- **GitOps:** Створено маніфест `argo-app.yaml`. Після встановлення Argo CD в кластер, він буде автоматично синхронізувати стан кластера з твоїм GitHub-репозиторієм. Будь-який `git push` у папку `k8s/` призведе до негайного оновлення застосунку на сервері.

## 5. Моніторинг (Prometheus + Grafana)
- **Full Stack:** Розгорнуто у окремому неймспейсі `monitoring`.
- **Prometheus:** Налаштовано Service Discovery для автоматичного збору метрик з усіх компонентів застосунку.
- **Grafana:** Попередньо налаштована для візуалізації стану кластера та метрик Python-скриптів.
